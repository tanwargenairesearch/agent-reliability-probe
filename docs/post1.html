Capable isn't the same as dependable

An AI agent that's right 85% of the time can still be dependable only 71% of the time. Both numbers are real — they just answer different questions. Here's what, why, and how.

tanwargenairesearch · 2026 · ~6 min read

Model scorecards tell you how capable a model is. But enterprises don't deploy a model — they deploy an agent: at least a model wrapped in tools, memory, and a workflow, and often more. The scorecard measures one part; production depends on the whole system.

The question scorecards aren't built for

Accuracy asks: can the agent do the task? That's exactly the right question for comparing models, and nothing here disputes it.

But for enterprise agents that take real actions — issuing a refund, updating a record, moving money, filing a ticket, executing a workflow step — there's a complementary question accuracy isn't designed to answer: not "can it do this once?" but "will it do it every time?"

In production, a task that succeeds 4 runs out of 5 isn't 80% done. It's an outcome you can't yet depend on. So alongside accuracy, we must measure consistency.

The two numbers

We took two recent frontier models, ran each as a customer-service agent on 100 real tasks, and ran every task 5 times.

  • Accuracy = pass@1 — succeeds at least once
  • Reliability = pass^5 — succeeds all five times

~71%

Both numbers are real, and both matter. Accuracy says the capability is there; reliability says how often you can count on it end-to-end. For workflows where every run reaches a customer, that ~14-point gap is the distance between a strong demo and a dependable deployment.

The dangerous failures are the intermittent ones

Most of the shortfall wasn't outright failure — it was intermittent failure. About 1 in 5 tasks were inconsistent: the agent handled them correctly on some runs and got them wrong on others, with nothing changed but the run itself.

A task that fails every time breaks in testing, so you fix it before launch. An intermittent failure passes your demo and your QA, ships with your confidence, then fails for a real customer on the one run that happens to break. Consistent failures you find; intermittent ones find you.

This is exactly why reliability needs its own metric. An intermittent task counts as a success under accuracy ("succeeded at least once") but a failure under reliability ("succeeded every time"). The gap between the two numbers is the inconsistency.

How it was measured

  • Tasks: τ-bench retail (Sierra Research) — real multi-turn customer-service tasks with a simulated user and a tool environment.
  • Grading: the actual database end-state — did the order really end up cancelled / exchanged / refunded correctly? No LLM judge.
  • Scale: 100 tasks × 5 independent runs = 500 episodes per model. Each model runs as the agent; the provider is the only thing that changes.
  • Metric: pass^k (τ-bench) using the unbiased combinatorial estimator from the Codex paper. Reported with bootstrap 95% confidence intervals.

The two models are statistically tied — the confidence intervals overlap, so there's no reliability difference between them on this task set. The story here isn't which model wins; it's the gap that both share.

A note on rigor

At 20 tasks, the two models looked clearly different with almost no reliability gap. At 100 tasks, they were tied and the gap was clear — same method, just enough samples to see it. A lesson for me as much as anyone: measure reliability on enough tasks before trusting either the ranking or the gap.
